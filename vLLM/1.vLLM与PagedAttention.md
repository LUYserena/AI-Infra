# vLLM 与 PagedAttention

## 为什么需要 vLLM？

大模型推理的核心挑战是 **KV Cache 的内存管理**。

在自回归生成过程中，每个 token 的生成都需要用到之前所有 token 的 Key 和 Value 向量。为了避免重复计算，推理时会把这些 K/V 存下来，这就是 **KV Cache**。

问题在于：**KV Cache 的大小在生成完成前是未知的**。

### 传统方法的问题

传统推理框架（如 HuggingFace Transformers）的做法：
- 在请求开始时，按 **最大序列长度**（max_seq_len）**预分配**一块连续显存
- 实际生成可能只用了一半，但剩下一半也被占着

```
预分配显存块（max_seq_len = 2048）

[已用 KV Cache 512 token] [空着但被占用 1536 token]
████████░░░░░░░░░░░░░░░░░░░░░░
```

这带来两个严重问题：

| 问题 | 说明 |
|------|------|
| **内部碎片** | 预分配空间用不完，显存被浪费 |
| **外部碎片** | 不同请求的显存块大小不一，中间有空洞塞不进新请求 |

实测：传统方法显存利用率只有 **20%~40%**，大量显存在空转。

## PagedAttention：用 OS 分页思想解决显存碎片

vLLM 的核心创新 **PagedAttention** 直接借鉴了操作系统的**虚拟内存分页**机制。

### 核心思路

把 KV Cache 切成固定大小的 **Block（页）**，按需分配，不再预分配一整块。

```
Block 大小 = 16 tokens（可配置）

请求A的 KV Cache：
Block 7 → Block 2 → Block 15 → ...（物理上不连续，逻辑上连续）

请求B的 KV Cache：
Block 3 → Block 9 → ...
```

类比：
- 传统方法 = 给每个请求订一个固定大小的包间，哪怕只来1个人也要占着
- PagedAttention = 共享大厅，来多少人用多少桌，桌子可以拼

### Block Table（块表）

每个请求维护一张 **Block Table**，记录逻辑 Block → 物理 Block 的映射，和 OS 的页表一个原理。

```
逻辑Block编号   物理Block编号
     0      →      7
     1      →      2
     2      →      15
```

Attention 计算时，vLLM 的 CUDA kernel 根据 Block Table 去不连续的物理地址取 K/V，**不需要物理连续**。

### 显存利用率提升

| 方法 | 显存利用率 |
|------|-----------|
| 传统预分配 | 20%~40% |
| PagedAttention | **接近 100%** |
| 吞吐量提升 | 最高 **24x**（论文数据） |

## vLLM 整体架构

vLLM 的核心由两部分组成：**LLM Engine（调度侧）** 和 **Worker（执行侧）**。

```
┌─────────────────────────────────────────┐
│              LLM Engine                  │
│                                          │
│  ┌──────────┐      ┌──────────────────┐  │
│  │ Scheduler│      │  Block Manager   │  │
│  │ 决定哪些 │      │  管理物理Block的  │  │
│  │ 请求跑   │      │  分配和释放      │  │
│  └──────────┘      └──────────────────┘  │
└─────────────────────────────────────────┘
              ↓  发送 block table + 输入
┌─────────────────────────────────────────┐
│              Worker（GPU侧）             │
│                                          │
│  ┌──────────────────────────────────┐    │
│  │     Model（含 PagedAttention）    │    │
│  │  根据 block table 执行 forward    │    │
│  └──────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### Scheduler（调度器）

调度器每个 step 决定：
- **哪些请求**进入本次 forward（prefill 或 decode 阶段）
- **分配多少 Block** 给这些请求
- **哪些请求需要 swap**（显存不够时换到 CPU）或 **recompute**

三种请求状态：
- `waiting`：排队等待，还没有 Block
- `running`：正在生成，已分配 Block
- `swapped`：显存不够，KV Cache 被换到 CPU

### Block Manager（块管理器）

- 维护一个空闲 Block 池
- 请求来了 → 从池里分配 Block → 写入 block table
- 请求完成 → Block 归还到池子
- 支持 **Copy-on-Write**：多个请求共享同一个 Block（prefix caching 的基础）

---

## Prefill 与 Decode 两个阶段

大模型推理分两个阶段，它们的特点完全不同：

| 阶段 | 说明 | 计算特点 |
|------|------|---------|
| **Prefill** | 一次性处理整个 prompt，生成第1个 token | 计算密集（compute-bound） |
| **Decode** | 每次只生成1个 token，自回归 | 内存密集（memory-bound） |

vLLM 在调度时需要平衡这两个阶段，这也是 **Chunked Prefill** 要解决的问题（下一篇展开讲）。

## 在 AI Infra 中的应用

PagedAttention 解决的是**推理吞吐量**的核心瓶颈，在工业部署中至关重要：

- **大规模服务**：同时处理数千并发请求，Block-level 管理让显存不再是瓶颈
- **长上下文推理**：GPT-4 128K、Gemini 1M token 场景下，KV Cache 巨大，PagedAttention 是必选项
- **多 LoRA 服务**：不同用户用不同 LoRA，vLLM 支持 Block 级别的 LoRA KV Cache 隔离
- **prefix caching**：系统 prompt 相同的请求共享 Block，直接省掉 prefill 计算（下篇讲）

💡 **核心结论**：vLLM 不是"更好的推理实现"，它是**重新设计了推理系统的内存管理模型**，把 OS 的分页思想搬到了 GPU 显存管理上。这是它能做到 24x 提升的根本原因。

## 面试高频问题

**Q1：vLLM 为什么比 HuggingFace 推理快？**
> 核心是 PagedAttention 解决了 KV Cache 的显存碎片问题，让显存利用率从 20-40% 提升到接近 100%，同一块显存能跑更多并发请求，吞吐量自然大幅提升。

**Q2：PagedAttention 和操作系统分页有什么关系？**
> 本质上是同一个思想：把连续逻辑地址映射到不连续物理地址，通过 Block Table（类比页表）管理映射关系。KV Cache 不再需要物理连续存储，碎片问题迎刃而解。

**Q3：vLLM 的 Scheduler 是怎么工作的？**
> 每个推理 step，Scheduler 决定哪些请求参与 forward（prefill/decode），调用 Block Manager 分配/回收 Block，生成 block table 下发给 GPU Worker 执行。显存不够时触发 swap 或 recompute。
