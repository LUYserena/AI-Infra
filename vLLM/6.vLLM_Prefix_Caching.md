# vLLM Prefix Caching 详解

## 什么是 Prefix Caching

**Prefix Caching（前缀缓存）** 是大模型推理中的一种优化技术：
- 缓存**请求共享的前缀部分**的 KV Cache
- 后续请求如果前缀相同，直接复用缓存，跳过重复计算

### 典型场景

1. **聊天机器人**
   ```
   请求1: [系统提示1500token] + "你好"
   请求2: [系统提示1500token] + "今天天气怎么样"
   请求3: [系统提示1500token] + "写一首诗"
   ```

2. **RAG（检索增强生成）**
   ```
   请求1: [文档A 3000token] + "总结一下"
   请求2: [文档A 3000token] + "翻译成英文"
   请求3: [文档B 2500token] + "提取关键信息"
   ```

3. **代码助手**
   ```
   请求1: [编程规范1000token] + "写一个快速排序"
   请求2: [编程规范1000token] + "修复这个bug"
   ```

## 一、为什么需要 Prefix Caching

### 重复计算的代价

假设系统提示 2000 token，单 A100 的 prefill 速度约 40 token/ms：
- 计算一次：2000 ÷ 40 = 50ms
- 计算 1000 次：50ms × 1000 = 50,000ms = 50秒

**50秒的 GPU 时间全在算同样的东西！**

### 经济成本

按 AWS p4d.24xlarge（8×A100）每小时 $32.77 计算：
- 50秒 GPU 时间价值：$32.77 × (50/3600) ≈ $0.455
- 日请求 100 万次：$0.455 × 1000 = $455/天
- 月浪费：$455 × 30 = $13,650

**每月浪费 1.3 万美元，就为了算同样的系统提示。**

## 二、Prefix Caching 原理

### 核心思想

利用 vLLM 的 **PagedAttention 机制**，把共享前缀的 KV Cache 存到 Block 中，多个请求复用。

### 工作流程

```
新请求到达
    ↓
1. 提取前缀部分（如系统提示）
    ↓
2. 计算前缀 hash
    ↓
3. 查缓存表：hash → Block IDs
    ↓
4. 如果命中：直接复用这些 Block
    ↓
5. 如果未命中：正常计算，并缓存结果
```

### hash 设计

前缀的 hash 需要：
- **快速计算**：不能成为性能瓶颈
- **低碰撞率**：不同前缀不能有相同 hash
- **稳定**：相同输入永远产生相同 hash

vLLM 使用 **MurmurHash3**，平衡速度和碰撞率。

## 三、实现细节

### 1. 缓存数据结构

```python
class PrefixCache:
    def __init__(self):
        # hash -> 对应的Block列表
        self.cache: Dict[int, List[Block]] = {}
        
        # Block -> 引用计数（多少个请求在用）
        self.ref_count: Dict[Block, int] = {}
```

### 2. Block 共享机制

关键问题：**多个请求共享 Block，一个请求修改了怎么办？**

解决方案：**Copy-on-Write（写时复制）**

```python
def share_blocks(request, prefix_hash):
    # 1. 从缓存获取Block列表
    blocks = self.cache[prefix_hash]
    
    # 2. 给请求分配这些Block（只读）
    request.blocks = blocks
    
    # 3. 增加引用计数
    for block in blocks:
        self.ref_count[block] += 1
        
def write_to_block(block, new_kv):
    # 如果需要写（比如生成了新token）
    if self.ref_count[block] > 1:
        # 复制一份新的Block
        new_block = block.copy()
        new_block.write(new_kv)
        
        # 原Block引用计数减1
        self.ref_count[block] -= 1
        
        return new_block
    else:
        # 只有自己在用，直接写
        block.write(new_kv)
        return block
```

### 3. 缓存失效策略

缓存不能无限增长，需要淘汰策略：

1. **LRU（最近最少使用）**：淘汰最久未用的前缀
2. **LFU（最不经常使用）**：淘汰使用频率最低的前缀
3. **TTL（生存时间）**：超过一定时间自动淘汰

vLLM 默认用 **LRU**，实现简单且效果好。

## 四、性能收益分析

### 理论收益

假设：
- 前缀长度：L 个 token
- 命中率：H（0-1）
- 单 token prefill 耗时：T

**不开缓存**：每个请求都要算 L×T
**开缓存**：只有 (1-H) 的请求要算 L×T

**加速比** = 1 / (1 - H + H/L) ≈ 1 / (1 - H)  （当 L 很大时）

### 实际测试数据

| 场景 | 前缀长度 | 命中率 | TTFT 降低 | 吞吐提升 |
|------|---------|--------|-----------|----------|
| 聊天机器人 | 1500 token | 95% | **85%** | 3.2x |
| RAG 系统 | 3000 token | 80% | **75%** | 2.5x |
| 代码助手 | 800 token | 90% | **82%** | 2.8x |
| 完全随机 | 0-500 token | 5% | 基本无 | 基本无 |

### 命中率的影响

```
命中率 vs 加速比（前缀2000token）
100% → ∞倍（理想情况）
90% → 10倍
80% → 5倍  
50% → 2倍
30% → 1.43倍
10% → 1.11倍
0% → 1倍（无效果）
```

**结论**：命中率 >50% 时收益显著，>80% 时效果极佳。

## 五、使用建议与最佳实践

### 1. 什么场景适合开

✅ **强烈推荐**：
- 固定系统提示的聊天机器人
- 文档固定的 RAG 系统
- 编程规范/风格固定的代码助手
- 任何前缀重复率高的场景

❌ **不太适合**：
- 每个请求前缀都不同
- 前缀很短（<200 token），收益有限
- 内存极其受限的环境

### 2. 参数调优

```python
# vLLM 配置示例
from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-2-7b-chat-hf",
    enable_prefix_caching=True,  # 开启缓存
    prefix_cache_max_capacity=100,  # 最多缓存100个前缀
    prefix_cache_ttl=3600,  # 1小时过期
)
```

**关键参数**：
- `prefix_cache_max_capacity`：缓存容量，根据内存调整
- `prefix_cache_ttl`：缓存存活时间，防止过时
- `block_size`：PagedAttention 的 block 大小，影响缓存粒度

### 3. 监控指标

部署后要监控：
- **命中率**：核心指标，反映缓存效果
- **内存占用**：缓存用了多少显存
- **TTFT 分布**：P50/P90/P99 首 token 时间
- **缓存淘汰率**：LRU 淘汰频率

## 六、在 AI Infra 中的应用

### 多租户 SaaS 服务

- **租户隔离**：每个租户有自己的系统提示，缓存按租户分区
- **成本分摊**：高频租户的缓存命中率高，成本更低

### 微服务架构

- **专用缓存服务**：独立部署 prefix cache 服务，多个推理服务共享
- **缓存预热**：服务启动时预加载常用前缀

### 边缘计算

- **资源受限环境**：显存宝贵，prefix caching 减少重复计算就是省钱
- **离线场景**：缓存常用前缀，断网时也能快速响应

### A/B 测试

- **多版本提示词**：不同版本的提示词分别缓存，对比效果
- **灰度发布**：新提示词逐步替换旧提示词，缓存平滑过渡

### 安全与合规

- **敏感信息隔离**：不同安全级别的提示词缓存隔离
- **数据驻留**：缓存数据不跨区域，满足合规要求

## 面试高频问题

**Q1：Prefix Caching 和普通的 KV Cache 有什么区别？**
> KV Cache 是单个请求内部缓存已生成的 token，用于自回归生成。Prefix Caching 是跨请求缓存共享的前缀（如系统提示），让不同请求复用相同的计算结果，避免重复计算。

**Q2：多个请求共享 Block 时，如何保证安全性（一个请求不会读到另一个请求的数据）？**
> 通过 Copy-on-Write 机制：Block 初始状态是只读共享的，当某个请求需要写入新 token 时，系统会复制一份新的 Block 给该请求，原 Block 继续给其他请求使用。这样既实现了共享，又保证了隔离。

**Q3：Prefix Caching 的命中率受哪些因素影响？如何提高命中率？**
> 影响因素：1) 请求前缀的重复程度；2) 缓存容量；3) 淘汰策略。提高方法：1) 设计更通用的系统提示；2) 增加缓存容量；3) 使用更智能的缓存策略（如预加载高频前缀）；4) 请求路由（相似前缀路由到同一实例）。

**Q4：什么场景下 Prefix Caching 效果最显著？什么场景下效果有限？**
> 最显著：固定系统提示的聊天机器人、文档固定的 RAG、代码助手等前缀重复率高的场景。效果有限：每个请求前缀都不同、前缀很短（<200 token）、请求模式完全随机的场景。
