# vLLM 调度策略详解

## 调度的重要性

在大模型推理服务中，**调度策略的好坏直接决定性能上限**。

- 显存管理（PagedAttention）解决了**能不能跑**的问题
- 调度策略解决了**跑得快不快、稳不稳**的问题

一个糟糕的调度器，能让 A100 跑出 1060 的效果。

---

## 一、Continuous Batching（连续批处理）

### 传统静态 Batching 的缺陷

早期推理框架使用**静态 Batching**：

```python
# 伪代码
batch = [request1, request2, request3, request4]  # 凑满batch
outputs = model.generate(batch)                   # 一次forward
# 等所有4个请求都生成完，才开始下一批
```

问题：
- **请求长度不均**：request1 生成了 5 个 token 就结束了，但要等其他 3 个
- **GPU 空转**：request1 结束后，它的 GPU slot 空着，但 batch 还没结束
- **延迟不公平**：短请求被长请求拖累

### Continuous Batching 的实现

vLLM 采用**动态 Batching**：

```python
# 每个step都重新组batch
def schedule_step():
    # 1. 检查哪些请求结束了，释放资源
    finished_requests = check_finished()
    release_resources(finished_requests)
    
    # 2. 从等待队列取新请求
    new_requests = get_new_requests()
    
    # 3. 组成新的batch（包含running的+新来的）
    current_batch = running_requests + new_requests
    
    # 4. 执行
    outputs = model.generate(current_batch)
```

**关键点**：batch 大小在每个 step 都可能变化，请求结束就离开，新请求随时加入。

### 性能对比

| 指标 | 静态 Batching | Continuous Batching |
|------|-------------|-------------------|
| GPU 利用率 | 40-60% | **85-95%** |
| 吞吐量（QPS） | 基准 | **2-10倍** |
| 平均延迟 | 高（被拖累） | 较低（相对公平） |
| 长尾延迟 | 极高 | 可控 |

## 二、Chunked Prefill（分块预填充）

### 问题：Prefill-Decode 干扰

大模型推理有两个阶段：

1. **Prefill（预填充）**：一次性处理整个 prompt，生成第一个 token
   - 计算密集（Compute-bound）
   - 耗时正比于 prompt 长度

2. **Decode（解码）**：自回归生成后续 token
   - 带宽密集（Memory-bound）
   - 每步很快，但要循环很多次

**干扰场景**：
- 系统正在服务 10 个 decode 请求（每步 20ms）
- 来了一个 8K token 的长 prompt（prefill 要 500ms）
- 如果让 prefill 独占执行：那 10 个 decode 请求**被卡住 500ms**

### Chunked Prefill 的解决方案

把长 prompt 的 prefill **切成固定大小的 chunk**，每个 step 只处理一个 chunk。

```python
# 假设 chunk_size = 512
prompt_length = 8192  # 8K token
num_chunks = ceil(8192 / 512) = 16

# 调度过程
for chunk_idx in range(num_chunks):
    # 每个step：decode请求 + 一个prefill chunk
    batch = decode_requests + [prefill_chunk[chunk_idx]]
    execute(batch)
```

### 权衡

- **优点**：decode 请求不再被长 prefill 卡死，延迟稳定
- **代价**：长 prompt 的**首 token 时间（TTFT）变长**
  - 原来：500ms 出第一个 token
  - 现在：16×20ms = 320ms + 一些 overhead

### chunk_size 调优

- **太大**（如 2048）：prefill 还是太占资源，干扰明显
- **太小**（如 128）：overhead 大，TTFT 变长很多
- **推荐值**：512（vLLM 默认），根据实际负载 profiling 调整

## 三、调度决策流程：vLLM v0 vs vLLM v1

vLLM 的调度器在 v0 和 v1 版本中有**本质区别**。理解这两个版本的调度差异，才能知道 vLLM 的演进方向。

### vLLM v0：阶段分离调度器

v0 调度器严格区分 **Prefill（预填充）** 和 **Decode（解码）** 两个阶段：

#### 架构特点
- **双阶段分离**：Prefill 和 Decode 被视为不同的计算模式
- **条件性 Chunked Prefill**：长 prompt 可被分块，但需手动开启
- **Swapping 机制**：显存不足时，KV Cache 可换出到 CPU 内存

#### 调度决策流程（v0）

```python
# 伪代码：v0 调度逻辑（简化）
def schedule_step_v0():
    # 1. 分离处理 Prefill 和 Decode 请求
    prefill_requests = filter_prefill(waiting_queue)
    decode_requests = filter_decode(running_queue)
    
    # 2. 优先调度 Prefill（保证 TTFT）
    if prefill_requests:
        batch = select_prefill(prefill_requests)
        # 可能独占 GPU，阻塞 Decode
        execute_prefill(batch)
    
    # 3. 再调度 Decode
    if decode_requests:
        batch = select_decode(decode_requests)
        execute_decode(batch)
    
    # 4. 状态转移：Prefill 完成后转为 Decode
    move_finished_prefill_to_decode()
```

#### v0 流程图

```
新请求到达 → Waiting队列（标记为Prefill）
     ↓
调度周期开始
     ↓
1. 检查Swapped队列 → 有空间则换入
     ↓  
2. 优先选择Prefill请求 → 保证首token延迟
     ↓
3. 分配Block资源 → 检查prefix cache
     ↓
4. 组装Batch → 可能独占GPU（长prefill）
     ↓
5. GPU执行 → Prefill完成后转为Decode
     ↓
6. Decode请求组batch执行 → Continuous Batching
     ↓
回到步骤1，下一个调度周期
```

**v0 的问题**：
1. **阶段干扰**：长 Prefill 会阻塞 Decode 请求
2. **复杂度高**：需协调两个阶段的资源分配
3. **功能集成难**：Chunked Prefill、Speculative Decoding 等需特殊处理

---

### vLLM v1：统一调度器（Unified Scheduler）

v1 调度器是 vLLM 架构重构的核心，采用 **Token 统一调度** 模型：

#### 架构特点
- **统一视图**：不再区分 Prefill 和 Decode，所有请求按 token 消耗统一管理
- **动态配额分配**：每个请求分配固定的 token 配额（包括 prompt 处理和生成）
- **默认 Chunked Prefill**：长 prompt 自动分块，无需特殊配置
- **移除 Swapping**：简化架构，不再需要 KV Cache 换入换出

#### 调度决策流程（v1）

```python
# 伪代码：v1 调度逻辑（简化）
def schedule_step_v1():
    # 1. 统一视图：所有请求按 token 消耗管理
    requests = all_requests()  # 不区分 prefill/decode
    
    # 2. 为每个请求分配 token 配额
    quotas = assign_token_quotas(requests)
    # 示例：{request_id: 512}  # 每个请求本轮最多消耗512个token
    
    # 3. 根据配额组装统一 batch
    batch = []
    for req in requests:
        # 消耗配额处理 prompt 剩余部分或生成新 token
        tokens_to_process = min(quotas[req.id], req.remaining_tokens)
        batch.append((req, tokens_to_process))
    
    # 4. 统一执行（包含 prompt chunking 和 generation）
    execute_unified_batch(batch)
    
    # 5. 更新剩余 token 数，请求完成则移除
    update_remaining_tokens()
```

#### v1 流程图

```
新请求到达 → Waiting队列（统一视图）
     ↓
调度周期开始
     ↓
1. 为每个请求分配token配额 → 动态计算
     ↓  
2. 根据配额选择处理的token数 → 包含prompt和生成
     ↓
3. 组装统一batch → 自动应用Chunked Prefill
     ↓
4. GPU统一执行 → 不区分prefill/decode阶段
     ↓
5. 更新配额和剩余token数 → 请求完成则移除
     ↓
回到步骤1，下一个调度周期
```

### 核心对比

| 特性 | vLLM v0 | vLLM v1 |
|------|---------|---------|
| **调度模型** | 阶段分离（Prefill/Decode） | **统一调度**（Token 级别） |
| **Chunked Prefill** | 有条件启用，需配置 | **默认启用**，自动处理 |
| **资源管理** | 复杂（需协调两个阶段） | 简化（统一 token 配额） |
| **Swapping** | 支持（显存不足时） | **移除**（简化架构） |
| **功能集成** | 困难（需适配两个阶段） | 容易（统一接口） |
| **调度策略** | FCFS（默认） | FCFS + **优先级调度** |

### 演进意义

1. **性能提升**：v1 在长上下文场景中有显著性能提升（官方数据）
2. **简化架构**：统一调度器减少系统复杂性和技术债
3. **未来友好**：为 Speculative Decoding、动态批处理等高级功能提供更好基础

💡 **实践建议**：
- **新项目直接用 v1**：统一调度器更简单、性能更好
- **迁移注意事项**：v1 移除 Swapping，需确保显存足够
- **调优重点**：v1 关注 `token_quota` 参数（控制每个请求每步的处理量）

## 四、调度优化技巧

### 1. 延迟与吞吐的权衡

- **偏向吞吐**：让 batch 尽量大，GPU 利用率高，但单个请求延迟可能增加
- **偏向延迟**：batch 小，响应快，但 GPU 可能没喂饱

**实践建议**：设置 `max_batch_size` 和 `min_batch_size`，根据 SLA 调整。

### 2. 长尾延迟治理

长尾延迟（P99延迟）往往由少数长请求或资源竞争引起。

**应对策略**：
- **隔离队列**：长短请求分队列，避免互相影响
- **抢占式调度**：短请求可以抢占长请求的资源（复杂，vLLM 暂未实现）
- **限流**：拒绝明显超长的请求

### 3. 多GPU调度

多卡时调度更复杂：

1. **数据并行**：每个 GPU 有完整的模型副本，请求可以路由到任意卡
2. **模型并行**：模型切分到多卡，请求必须跨卡执行

vLLM 主要支持**数据并行**，调度器需要：
- 监控各 GPU 负载
- 均衡分配请求
- 处理 GPU 间通信

## 五、在 AI Infra 中的应用

### 高并发聊天服务

- **实时对话**：Continuous Batching 保证多人同时聊天不卡顿
- **流式输出**：调度器支持 token-by-token 流式返回

### 批量处理任务

- **文档总结**：Chunked Prefill 处理长文档时不阻塞其他请求
- **代码生成**：多个用户同时写代码，调度器公平分配 GPU 资源

### A/B 测试与多模型

- **流量切分**：调度器可以把请求路由到不同模型版本
- **灰度发布**：逐步增加新模型的流量比例

### 成本优化

- **闲时合并**：低峰期把小请求合并成大 batch，节省成本
- **自动扩缩容**：根据调度器负载指标，动态调整 GPU 数量

## 面试高频问题

**Q1：Continuous Batching 和静态 Batching 的主要区别是什么？**
> 静态 Batching 一次组一个固定 batch，等所有请求完成再开始下一批，GPU 有空窗期。Continuous Batching 是动态的，每个 step 都重新组 batch，请求结束就离开、新请求随时加入，GPU 持续满载，吞吐量提升 2-10 倍。

**Q2：什么是 Prefill-Decode 干扰？Chunked Prefill 如何解决？**
> 长 prompt 的 prefill 计算量大、耗时长，会阻塞同时在跑的 decode 请求，导致 decode 延迟飙升。Chunked Prefill 把长 prefill 切成固定大小的 chunk，每个 step 只处理一个 chunk，和 decode 请求混跑，避免独占 GPU。

**Q3：vLLM 调度器选择请求时有哪些策略？各有什么优劣？**
> 主要有三种：1) FCFS（先来先服务）：公平但可能被长请求拖累整体延迟；2) SJF（最短作业优先）：降低平均延迟但可能饿死长请求；3) 优先级调度：支持业务需求但实现复杂。vLLM 默认用 FCFS，在公平性和实现复杂度间平衡。
