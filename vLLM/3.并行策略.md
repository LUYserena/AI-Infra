# 大模型并行策略全览：TP / PP / DP / EP / SP

## 为什么需要并行？

大模型的参数量已经超出单张 GPU 的承载能力：

- LLaMA-3 70B：FP16 下约 **140GB 显存**
- A100 80GB：单卡放不下
- 解法：把模型切开，分到多张卡上

但"切"的维度不同，就产生了不同的并行策略。

---

## 一、数据并行 Data Parallelism（DP）

### 核心思路

**模型不切，数据切。**

每张卡放一份完整的模型副本，不同的卡处理不同的数据（mini-batch），各自独立做 forward 和 backward，最后把梯度汇总（all-reduce）同步参数。

```
卡0: 模型副本 → batch[0:32]  → 梯度0 ─┐
卡1: 模型副本 → batch[32:64] → 梯度1 ─┤→ All-Reduce → 参数同步
卡2: 模型副本 → batch[64:96] → 梯度2 ─┘
```

### 适用场景
- 模型能放进单卡（参数量不太大）
- 想提升训练吞吐量
- PyTorch DDP 是最常见的实现

### 局限
- 模型必须能放进单卡显存
- 通信量 = 一份梯度的大小（随模型增大线性增加）

---

## 二、张量并行 Tensor Parallelism（TP）

### 核心思路

**把单个算子（矩阵）切开，分到多卡。**

以 Linear 层为例，权重矩阵 W 可以按列切（Column Parallel）或按行切（Row Parallel）：

```
原始: Y = X × W   （W 是一个大矩阵）

列切分（Column Parallel）:
  卡0: Y0 = X × W[:, :half]
  卡1: Y1 = X × W[:, half:]
  → 结果拼接: Y = [Y0, Y1]

行切分（Row Parallel）:
  卡0: Y0 = X[:, :half] × W[:half, :]
  卡1: Y1 = X[:, half:] × W[half:, :]
  → 结果相加（All-Reduce）: Y = Y0 + Y1
```

Megatron-LM 把 Transformer 里的 MLP 和 Attention 层都做了 TP 切分。

### 适用场景
- 单层参数量大，单卡放不下
- 推理场景最常用（vLLM 默认支持 TP）
- 要求卡间带宽高（NVLink 最佳）

### 局限
- 每层都需要卡间通信（All-Reduce）
- 通信频繁，对网络带宽要求高
- 扩展到太多卡时通信开销显著

---

## 三、流水线并行 Pipeline Parallelism（PP）

### 核心思路

**按层切，不同卡负责不同的层（Stage）。**

```
卡0: Layer 0-8   （Stage 0）
卡1: Layer 9-16  （Stage 1）
卡2: Layer 17-24 （Stage 2）
卡3: Layer 25-32 （Stage 3）
```

数据像工厂流水线一样流过：卡0 处理完把激活值传给卡1，卡1 传给卡2……

### 流水线气泡问题

朴素实现里，卡0 处理完等卡1，卡1 处理完等卡2……大多数时间在空等，这叫**流水线气泡（Pipeline Bubble）**。

```
朴素实现:
卡0: [F0][F0][F0][F0]         [B0][B0][B0][B0]
卡1:          [F1][F1][F1][F1]         [B1]...
（气泡占比 = (p-1)/p，p 是 stage 数）
```

**1F1B（One Forward One Backward）** 调度可以大幅减少气泡，GPipe / PipeDream 等工作专门优化这个问题。

### 适用场景
- 模型层数多，单卡放不下所有层
- 卡间带宽不够高（PP 通信量远少于 TP）
- 训练大模型时与 TP 结合使用

### 局限
- 存在流水线气泡，GPU 利用率有损失
- 需要精心设计每个 Stage 的层数（负载均衡）

---

## 四、专家并行 Expert Parallelism（EP）

### 核心思路

**MoE 模型专属。把不同的 Expert（专家网络）放到不同的卡上。**

MoE 模型结构回顾：每个 token 由 Router 选择 Top-K 个 Expert 处理，而不是走全部参数。

```
Token → Router → 选择 Expert 2 和 Expert 7

卡0: Expert 0, Expert 1, Expert 2, Expert 3
卡1: Expert 4, Expert 5, Expert 6, Expert 7

→ Token 需要的 Expert 2 在卡0，Expert 7 在卡1
→ Token 要分别发给两张卡 → All-to-All 通信
```

### 核心挑战

| 挑战 | 说明 |
|------|------|
| **All-to-All 通信** | 每个 token 要找到自己对应的 Expert，通信模式复杂 |
| **负载不均衡** | Router 可能把大量 token 都路由到同一个 Expert（热点问题） |
| **Expert 容量限制** | 每个 Expert 每步能处理的 token 数有上限，超出就丢弃（token dropping） |

DeepSeek、Mixtral 等 MoE 模型大量使用 EP，负载均衡是核心优化方向之一。

### 适用场景
- MoE 架构模型（Mixtral、DeepSeek-V2/V3、Qwen3-MoE 等）
- Expert 数量多、单卡放不下所有 Expert

---

## 五、序列并行 Sequence Parallelism（SP）

### 核心思路

**把序列（token 维度）切开，不同卡处理不同位置的 token。**

当序列长度极长时（如 128K、1M token），单卡的 Attention 计算和 KV Cache 存不下，这时需要 SP。

```
序列长度 = 128K tokens

卡0: token [0, 32K)   → 计算这段的 Attention
卡1: token [32K, 64K) → 计算这段的 Attention
卡2: token [64K, 96K) → 计算这段的 Attention
卡3: token [96K, 128K)→ 计算这段的 Attention
```

由于 Attention 需要 token 之间互相看到（全局 Attention），卡之间需要做 K/V 的通信（Ring Attention 是一种经典实现）。

### 适用场景
- 超长序列训练/推理（>32K token）
- 通常与 TP 结合使用（Megatron-LM 中 SP 和 TP 耦合在一起）

---

## 混合并行：3D 并行

实际训练大模型时，单种并行不够用，通常混合使用：

```
典型 3D 并行配置（以 LLaMA-3 70B 训练为例）：

DP × TP × PP

DP=8:  8 路数据并行（8 个 batch）
TP=4:  每个模型切成 4 份（张量并行）
PP=4:  模型按层切成 4 个 Stage

总卡数 = 8 × 4 × 4 = 128 张 GPU
```

MoE 模型还会额外加入 EP：**DP × TP × PP × EP = 4D 并行**

---

## 五种并行对比总结

| 并行策略 | 切分维度 | 通信操作 | 适用场景 | 通信量 |
|---------|---------|---------|---------|-------|
| **DP** | 数据（batch） | All-Reduce（梯度） | 模型能放单卡，提升吞吐 | 中 |
| **TP** | 参数（矩阵列/行） | All-Reduce（每层） | 单层太大，推理常用 | 高（频繁） |
| **PP** | 模型层 | P2P（层间激活） | 层数多，卡间带宽低 | 低 |
| **EP** | Expert 分配 | All-to-All | MoE 模型 | 高（不规则） |
| **SP** | 序列长度 | All-Gather / Ring | 超长序列 | 中高 |

## 在 AI Infra 中的应用

- **vLLM 推理**：默认使用 TP，`tensor_parallel_size=N` 即可开启 N 卡张量并行
- **Megatron-LM 训练**：3D 并行（DP+TP+PP）的标准实现，大规模训练主流方案
- **DeepSeek / Mixtral 推理**：EP 是核心，Expert 路由的 All-to-All 通信优化是性能关键
- **长上下文模型**：Kimi、Gemini 等支持百万 token 的模型，训练时需要 SP

💡 **选并行策略的经验法则**：
1. 模型能放单卡 → 先用 DP
2. 单层放不下 → 加 TP（但卡间带宽要够）
3. 层数太多单卡放不下 → 加 PP
4. MoE 模型 → 必须上 EP
5. 序列超长 → 加 SP

## 面试高频问题

**Q1：张量并行和流水线并行的区别是什么？**
> TP 是把单个算子（矩阵）切开分到多卡，每层都要通信，通信频率高但延迟低；PP 是按层切，不同卡负责不同层，通信只在层边界发生，通信量少但存在流水线气泡。大模型训练通常把两者结合使用。

**Q2：专家并行为什么需要 All-to-All 通信？**
> MoE 模型里每个 token 被 Router 路由到特定的 Expert，而 Expert 分布在不同的卡上。每张卡上的 token 可能需要发给任意其他卡上的 Expert 处理，因此是典型的 All-to-All 通信模式，与 DP 的 All-Reduce 不同。

**Q3：什么是流水线气泡？怎么缓解？**
> 在朴素流水线并行中，由于各 Stage 需要等前一个 Stage 完成才能开始，导致大多数卡处于空闲状态，浪费比例约为 (p-1)/p（p 为 Stage 数）。1F1B 调度策略通过交错 forward 和 backward，将气泡比例大幅压缩。
