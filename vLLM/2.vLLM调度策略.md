# vLLM 调度策略详解

## 回顾：推理的两个阶段

大模型推理分两个性质完全不同的阶段：

| 阶段 | 做什么 | 计算特点 | 耗时 |
|------|--------|---------|------|
| **Prefill** | 一次性处理整个 prompt，生成第 1 个 token | Compute-bound（计算密集） | 正比于 prompt 长度 |
| **Decode** | 每步只生成 1 个 token，自回归循环 | Memory-bound（带宽密集） | 每步很快，但要循环很多次 |

理解这两个阶段的差异，是理解所有调度策略的前提。

---

## 一、Continuous Batching（连续批处理）

### 传统静态 Batching 的问题

早期推理框架用**静态 Batching**：

```
Step 1: [请求A, 请求B, 请求C] → forward → 等所有请求都生成完
Step 2: [请求D, 请求E, 请求F] → forward → ...
```

问题：请求 A 生成了 5 个 token 就结束了，但系统要等 B 和 C 都结束才开始下一批。
A 结束后那个 slot 空着，GPU 在空转。

### Continuous Batching 的做法

vLLM 的做法：**哪个请求结束了，立刻把新请求填进来**，batch 大小动态变化。

```
Step 1: [A, B, C] → A生成完了
Step 2: [B, C, D] → D是新来的，立刻塞进来
Step 3: [B, C, D, E] → E又来了
...
```

GPU 永远是满的，不存在等待的空窗。

### 效果

| 指标 | 静态 Batching | Continuous Batching |
|------|-------------|-------------------|
| GPU 利用率 | 低（有空窗） | 高（接近满载） |
| 吞吐量 | 基准 | **提升 2-10x** |
| 延迟公平性 | 短请求被长请求拖累 | 相对公平 |

## 二、Chunked Prefill（分块预填充）

### 问题：Prefill 会把 Decode 卡死

假设系统里正在跑 10 个请求的 decode（每步很快），这时来了一个 prompt 超长的新请求（比如 8K token 的文档分析）。

如果让这个超长 prefill 整块执行：
- 这一个 step 要处理 8K token，**耗时极长**
- 期间那 10 个 decode 请求全部被阻塞
- 用户感知到的就是：**我的对话突然卡了好几秒**

这个问题叫 **Prefill-Decode 干扰（Interference）**。

### Chunked Prefill 的解法

把长 prompt 的 prefill 切成固定大小的 **chunk**，每个 step 只处理一个 chunk，和 decode 请求混在一起跑。

```
chunk_size = 512 token（可配置）

Step 1: [decode×10, prefill chunk 1/16]
Step 2: [decode×10, prefill chunk 2/16]
...
Step 16: [decode×10, prefill chunk 16/16] → 第1个token生成完毕
```

💡 代价：长 prompt 的**首 token 时间（TTFT）变长**了，但 decode 请求不再被卡死，整体延迟更稳定。

### chunk_size 的调参影响

- **chunk_size 太大**：prefill 还是会抢占 decode，延迟抖动大
- **chunk_size 太小**：prefill 被切太碎，overhead 增加，TTFT 变得更长
- **推荐起点**：512，根据实际负载 profile 后调整

## 三、Prefix Caching（前缀缓存）

### 场景

很多推理请求共享**相同的系统提示（system prompt）**：

```
请求1：[系统提示 2000 token] + [用户问题A 50 token]
请求2：[系统提示 2000 token] + [用户问题B 80 token]
请求3：[系统提示 2000 token] + [用户问题C 30 token]
```

如果每个请求都重新 prefill 一遍系统提示，是巨大的浪费。

### Prefix Caching 的做法

利用 PagedAttention 的 Block 机制：

- 系统提示的 KV Cache 存到若干 Block 里
- 这些 Block 被**多个请求共享**（Copy-on-Write 保证安全）
- 新请求来了，先检查前缀 Block 有没有缓存 → **命中直接复用，不用重算**

```
Block hash: hash(系统提示 tokens) → Block ID

请求1 → 检查 hash → 命中 → 跳过 2000 token 的 prefill
请求2 → 检查 hash → 命中 → 跳过 2000 token 的 prefill
请求3 → 检查 hash → 命中 → 跳过 2000 token 的 prefill
```

### 效果

| 场景 | 命中率 | TTFT 降幅 |
|------|--------|---------|
| 固定系统提示的 chatbot | 极高（>90%） | 显著，正比于系统提示长度 |
| RAG 场景（文档固定） | 高 | 显著 |
| 完全随机请求 | 低 | 基本无收益 |

💡 **实际部署建议**：如果你的系统提示很长（>1K token），一定要开 prefix caching，这是免费的推理加速。

## 调度器决策流程（完整版）

每个推理 step，vLLM Scheduler 的决策逻辑：

```
1. 从 waiting 队列里取新请求
   ↓
2. 检查 prefix cache → 有命中就跳过对应 prefill
   ↓
3. 判断是否做 chunked prefill（prompt 是否超过 chunk_size）
   ↓
4. 和正在 running 的 decode 请求合并成一个 batch
   ↓
5. 分配 Block → 生成 block table → 下发给 Worker 执行
   ↓
6. 收到输出 → 判断是否结束 → 结束则释放 Block，腾出空间
```

## 在 AI Infra 中的应用

- **高并发 API 服务**：Continuous Batching 让单卡能服务数百并发，是 vLLM 商业部署的核心优势
- **长文档 RAG**：Chunked Prefill + Prefix Caching 组合，大幅降低长 context 推理延迟和成本
- **多轮对话系统**：历史对话 KV Cache 可以通过 prefix caching 复用，降低每轮的 TTFT
- **Agent 工具调用**：Agent 每步都要推理，Continuous Batching 保证多 Agent 并发时 GPU 不空转

💡 **核心结论**：vLLM 的快不是单靠 PagedAttention，而是 **PagedAttention（显存）+ Continuous Batching（吞吐）+ Chunked Prefill（延迟稳定）+ Prefix Caching（重复计算）** 四个机制协同工作的结果。

## 面试高频问题

**Q1：Continuous Batching 和静态 Batching 的区别是什么？**
> 静态 Batching 一次跑一批，等所有请求完成再开始下一批，GPU 有空窗。Continuous Batching 是动态的，一个请求结束立刻把新请求填入，GPU 持续满载，吞吐量提升 2-10x。

**Q2：什么是 Prefill-Decode 干扰？Chunked Prefill 怎么解决？**
> 长 prompt 的 prefill 一次要处理大量 token，耗时长，会阻塞同时在跑的 decode 请求，导致延迟抖动。Chunked Prefill 把 prefill 切成固定大小的 chunk，每步只处理一小块，和 decode 混跑，避免独占 GPU。

**Q3：Prefix Caching 的原理是什么？什么场景下最有效？**
> 利用 PagedAttention 的 Block 机制，对共享前缀（如系统提示）的 KV Cache 做 hash 索引，相同前缀的请求直接复用已缓存的 Block，跳过重复 prefill。在系统提示固定的 chatbot 和 RAG 场景下效果最显著。
