# AI Infra —— 体系结构篇 · 知识全景总结

## 一、为什么AI Infra需要学体系结构

大模型推理/训练的性能瓶颈，归根结底就是三件事：

```
计算瓶颈 ← 需要理解CPU/GPU的计算架构
    ↓
存储瓶颈 ← 需要理解Cache层次、内存管理
    ↓
通信瓶颈 ← 需要理解进程通信、并行模型
```

体系结构是AI Infra的地基，地基不牢，上层优化无从谈起。

## 三、CPU vs GPU 核心对比

| 维度 | CPU | GPU |
|------|-----|-----|
| 设计目标 | 低延迟、复杂逻辑 | 高吞吐、大规模并行 |
| 核心数 | 几个到几十个（强核心） | 数千到上万个（弱核心） |
| 并行模型 | SIMD（SSE/AVX） | SIMT（Warp） |
| 适合任务 | 分支多、逻辑复杂的任务 | 数据并行度高的任务（如矩阵运算） |
| Cache | 大（MB级），命中率优先 | 小（KB级），吞吐优先 |
| 内存带宽 | ~100 GB/s | ~2-3 TB/s（HBM） |

> 💡 大模型推理为什么用GPU？因为核心运算是矩阵乘法（GEMM），数据并行度极高，GPU的大规模并行+高带宽完美匹配。

## 四、存储层次全景

```
速度快                                        容量大
←─────────────────────────────────────────────→

寄存器 → L1 Cache → L2 Cache → 主存(DRAM) → 磁盘(SSD)
 ~1ns     ~2ns      ~10ns      ~100ns       ~10μs

GPU侧：
寄存器 → 共享内存/L1 → L2 Cache → HBM(显存) → CPU主存(PCIe)
 ~1ns      ~5ns        ~50ns      ~200ns      ~10μs
```

> 💡 AI Infra的核心优化思路：**让数据尽量待在离计算最近的地方**。KV Cache放在HBM里，热数据尽量放共享内存。
